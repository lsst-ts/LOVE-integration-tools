1. LOVE Project Overview

The LSST Operation and Visualization Environment (L.O.V.E.) project consists in the development of graphical user interfaces (GUI) for the monitoring and operation of the LSST. The GUIs are built using web technologies in order to enable users access from different locations and different host systems.

This section provides a general overview of the LOVE project, how it is structured, how its software components interact, etc. For usage instructions please refer to the LOVE-integration-tools instructions section.

1.1. LOVE Software repositories

The LOVE project is integrated by the following software components, each with its own repository

1.1.1. LOVE-frontend

The LOVE-frontend is the web application that provides the LOVE GUIs. It is built using ReactJS to generate the static content (html, css and javascript files) of the LOVE web application. For more details check the LOVE-frontend docs.

1.1.2. LOVE-manager

The LOVE-manager is a backend application that acts as middleware for the LOVE-frontend. It is built using Django Channels with Redis as a Channel Layer For more details check the LOVE-manager docs.

1.1.3. LOVE-producer

The LOVE-producer is a backend application that acts as middleware between the LOVE-manager and SAL. It is built as a Python module that uses salobj and websockets to produce messages between SAL and the LOVE-manager. “Messages” can be understood without distinction as events or telemetries . For more details check the LOVE-producer docs.

1.1.4. LOVE-CSC

The LOVE-CSC is defined in the LOVE-producer repository, it is a CSC that provides an HTTP API to accept requests with observing logs messages and dispatches the observing logs event using salobj. For more details check the LOVE-producer docs.

1.1.5. LOVE-commander

The LOVE-commander is a backend application that acts as middleware between the LOVE-manager and SAL. It is built as a Python module that provides and HTTP API to accept command requests and run them using salobj, and provide some SAL information, such as metadata, topics names, among others. For more details check the LOVE-commander docs.

1.1.6. LOVE-simulator

The LOVE-simulator repository contains different Python modules used to simulate the software components of a real LSST environment interacting with SAL. It is meant to be used for development and testing purposes.

1.1.7. Database

The Database is used by the LOVE-manager to store users, their permissions and access tokens, the UI Framework views, among others. It is currently a PostgreSQL instance.

1.1.8. Channels Layer

The Channels Layer is used by the LOVE-manager as a message queue for the websockets messages. It is currently a Redis instance.

1.2. LOVE Architecture

As shown in the figure above, the LOVE-frontend is executed in the client browser (as it is a web application) while the rest of the components are executed in the LSST Servers. The LOVE-frontend communicates with the rest of the system through the LOVE-manager. The LOVE-frontend sends users credentials to login and then established a websocket connection with the LOVE-manager to receive data from it.

The LOVE-manager gets the data from the LOVE-Producer and LOVE-Commander through a websocket and HTTP. The LOVE-Producer and LOVE-Commander interacts with the rest of the system through the Software Abstraction Layer (SAL). The rest of the system consists on Commandable SAL Components (CSCs) which interact with each other through SAL.

Additionally, observing logs are sent from the LOVE-frontend to the LOVE-manager and from there to the LOVE-CSC thorugh websockets. The LOVE-CSC dispatches an event with the observing log message using salobj. The event then arrives to the LOVE-frontend through the same path as any other event, though the LOVE-producer and LOVE-manager,

CSCs communication consists in 4 types:

  • Telemetries: values that represent a software or hardware measurement in the real world

  • Events: that are triggered upon some defined conditions, for example, when a critical condition is reached, or when a process has finished.

  • Commands: instructions for another CSC to execute an action.

  • Command Acknowledgements: acknowledgements of sucessful command executions, sent for the CSC that sent the command.

We also developed the LOVE-simulator, for development and testing purposes, which acts as a “fake” CSC that sends data and commands to SAL, replicating the behavior of some defined CSCs.

1.3. LOVE Deployment

The figure above shows how the LOVE components are currently being deployed. They are all orquestrated with docker-compose and are present in the Docker network. Additionally, the components that interact with the SAL/DDS are also part of the DDS Network. This way they can interact with other CSC outside the Docker network.

It is worth to mention that different instances of the LOVE-producer are deployed, each of them takins care of different message, namely: telemetries, events, heartbeats, ScriptQueue state. For more deatils about this please refere to the LOVE-producer docs.

The user interacts with the LOVE system through a Nginx instance, that acts as load balancer and URL router. Routing requests to the LOVE-frontend and LOVE-manager accordingly.
